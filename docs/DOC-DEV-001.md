**О документе:** документ описывает архитектуру, компоненты, процесс обработки данных и инструкции по эксплуатации веб-приложения Tempa.

**Для кого:** для разработчиков, администраторов и пользователей, которые хотят понять внутреннее устройство и научиться запускать и сопровождать систему.

**Основано на:** на коде проекта, документации FastAPI, Celery, OpenCV и лучших практиках построения веб-приложений.

## 1. Архитектура приложения

Tempa построен на основе микросервисного подхода с выделением отдельных компонентов:

- **Бэкенд (FastAPI)** – REST API, обработка запросов, управление задачами, аутентификация, раздача статики.
- **Очередь задач (Celery + Redis)** – асинхронная обработка видео. Позволяет не блокировать интерфейс пользователя при длительных вычислениях.
- **База данных (SQLite)** – хранение информации о пользователях, задачах, результатах обработки.
- **Фронтенд (HTML + CSS + JS)** – одностраничное приложение для загрузки файлов, настройки параметров, отображения результата и интерактивной работы с порогом.
- **Процессор видео (OpenCV + Matplotlib)** – извлечение кадров, построение карты максимумов, рендеринг тепловой карты с цветовой шкалой.

### Схема взаимодействия компонентов

```
[Пользователь] → [Фронтенд] → [FastAPI] → [Celery] → [Redis]
                     ↑            ↓           ↓
                     └────────────┴───────────┘
                       [SQLite] ← [Результат]
```

1. Пользователь загружает файл через интерфейс.
2. Фронтенд отправляет запрос на `/tasks/upload` с файлом и параметрами.
3. Бэкенд сохраняет задачу в БД со статусом `pending` и отправляет задачу в Celery.
4. Celery worker обрабатывает видео, обновляет статус задачи и сохраняет результат в БД.
5. Фронтенд опрашивает статус задачи через `/tasks/status/{task_id}`.
6. После завершения отображается тепловая карта.
7. Пользователь может менять порог – отправляется запрос на `/tasks/recolor`, и карта пересчитывается без повторной обработки видео.

## 2. Компоненты и их функции

### 2.1. Бэкенд (FastAPI)

- **`main.py`** – точка входа, настройка CORS, middleware, подключение роутеров, раздача статики.
- **`api/auth.py`** – регистрация, логин, логаут, проверка сессии через подписанные куки.
- **`api/tasks.py`** – загрузка файлов, получение статуса задачи, пересчёт порога.
- **`core/config.py`** – настройки из переменных окружения.
- **`core/database.py`** – подключение к SQLite, создание сессий.
- **`core/security.py`** – хеширование паролей, создание и проверка сессионных токенов.
- **`core/dependencies.py`** – зависимости для роутеров (получение БД, текущего пользователя).
- **`models/task.py`** – модель задачи (поля: id, user_id, filename, параметры, статус, результат, ошибка).
- **`models/user.py`** – модель пользователя (id, username, password_hash).
- **`services/processor.py`** – основная логика обработки видео: открытие файла, извлечение кадров, построение карты максимумов, рендеринг тепловой карты, пересчёт порога.
- **`services/task_manager.py`** – создание, обновление и получение задач из БД.
- **`celery_app.py`** – настройка Celery (брокер Redis, сериализация, таймауты).
- **`tasks.py`** – Celery-задача `process_video_task`, которая вызывает `processor.process_video` и обновляет статус.

### 2.2. Фронтенд

- **`index.html`** – разметка страниц (логин, основная страница с загрузкой, результатом).
- **`style.css`** – стили (адаптивный дизайн, цветовая схема).
- **`script.js`** – логика: аутентификация, загрузка файла, опрос статуса, обновление порога, скачивание PNG.

### 2.3. Очередь задач (Celery + Redis)

- **Redis** – брокер и бекенд результатов.
- **Celery** – распределение задач между воркерами. Воркер запускается отдельным процессом и выполняет задачи асинхронно.

### 2.4. База данных (SQLite)

- Хранит пользователей и задачи. Файл `tempa.db` создаётся автоматически при первом запуске.
- Для продакшена рекомендуется переключиться на PostgreSQL или другую СУБД (изменив `DATABASE_URL`).

## 3. Процесс обработки данных

1. **Загрузка файла** – пользователь выбирает `.ravi` файл через интерфейс.
2. **Создание задачи** – бэкенд создаёт запись в БД со статусом `pending`.
3. **Отправка в Celery** – задача ставится в очередь.
4. **Обработка воркером**:
   - Открытие видео через OpenCV.
   - Извлечение всех кадров в градациях серого.
   - Вычисление максимальной яркости для каждого пикселя.
   - Преобразование яркости в температуру по линейной шкале (T_min – T_max).
   - Построение тепловой карты с использованием палитры RAVI и обрезанного по порогу диапазона.
   - Сохранение результата (base64 PNG) и матрицы максимумов (base64).
5. **Обновление статуса** – после успешной обработки статус меняется на `completed`, в БД сохраняется результат.
6. **Опрос статуса** – фронтенд периодически запрашивает статус задачи.
7. **Отображение результата** – картинка подставляется в тег `<img>`.
8. **Пересчёт порога** – при изменении ползунка фронтенд отправляет новый порог на `/tasks/recolor`, бэкенд пересчитывает карту из сохранённой матрицы и возвращает новую картинку.

## 4. Инструкция по эксплуатации

### 4.1. Запуск на локальном ПК

1. Установите Python 3.12+, Git, Redis.
2. Клонируйте репозиторий.
3. Создайте виртуальное окружение, установите зависимости.
4. Настройте `.env` (скопируйте из `.env.example`).
5. Запустите Redis.
6. Запустите Celery worker.
7. Запустите Uvicorn.
8. Откройте `http://127.0.0.1:8000/` в браузере.

Подробные команды – в `README.md`.

### 4.2. Запуск на сервере (Docker)

```bash
docker-compose up --build
```

После запуска приложение доступно по адресу сервера на порту 8000.

### 4.3. Настройка окружения

Обязательные переменные в `.env`:

| Переменная      | Описание                                      | Пример                       |
| --------------- | --------------------------------------------- | ---------------------------- |
| `SECRET_KEY`    | Секретный ключ для подписи кук                | `your-secret-key-here`       |
| `DATABASE_URL`  | URL для подключения к БД                      | `sqlite:///./tempa.db`       |
| `REDIS_URL`     | URL для подключения к Redis                   | `redis://localhost:6379/0`   |
| `TEMP_DIR`      | Временная папка для загруженных файлов        | `/tmp/tempa` (или `C:\Temp`) |
| `MAX_FILE_SIZE` | Максимальный размер загружаемого файла (байт) | `2147483648` (2 ГБ)          |

### 4.4. Обновление и миграции

При изменении схемы БД (моделей) необходимо удалить файл `tempa.db` (или использовать Alembic для миграций). В текущей версии миграции не реализованы.

### 4.5. Мониторинг и логи

- Логи бэкенда выводятся в терминал с Uvicorn.
- Логи Celery – в терминале с worker.
- Для продакшена рекомендуется настроить сбор логов в файл (через `logging`).

## 5. Безопасность

- **Аутентификация** – сессионные куки, подписанные `itsdangerous`. Пароли хранятся в хешированном виде (bcrypt).
- **CORS** – разрешены только доверенные источники (настраивается в `main.py`).
- **Защита от CSRF** – не реализована, но куки имеют атрибут `SameSite=Lax`.
- **SQL-инъекции** – SQLAlchemy использует параметризованные запросы.
- **XSS** – на фронтенде нет шаблонизации, все данные приходят через JSON и вставляются через `innerHTML` (риск минимален).

## 6. Возможные проблемы и их решение

| Проблема                  | Вероятная причина                 | Решение                                                                        |
| ------------------------- | --------------------------------- | ------------------------------------------------------------------------------ |
| 401 Unauthorized          | Кука не передаётся или истекла    | Выйдите и войдите заново. Проверьте, что используете `http://127.0.0.1:8000/`. |
| 500 Internal Server Error | Ошибка в обработке файла          | Проверьте логи. Убедитесь, что файл не повреждён и имеет расширение `.ravi`.   |
| Задача не обрабатывается  | Celery worker не запущен          | Запустите worker в отдельном терминале.                                        |
| Redis не отвечает         | Redis не запущен                  | Запустите Redis (через WSL, службу или Docker).                                |
| Файл не читается          | Неподдерживаемый формат или кодек | Попробуйте преобразовать файл в `.avi` или `.mp4` с помощью ffmpeg.            |

## 7. Расширение и доработка

- **Поддержка других форматов** – в `processor.py` можно добавить новые способы чтения (например, через `imageio`).
